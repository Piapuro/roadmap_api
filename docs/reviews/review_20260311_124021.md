## 🔴 Critical Issues

### [C-1] SignUp の 409 エラーが実装とSwagger定義で乖離

**ファイル**: `controller/auth_controller.go:40-42`
**問題**: Swagger godoc に `@Failure 409 "メールアドレスが既に登録済み"` と定義されているが、実装では `authService.SignUp` のエラーを区別せず常に 500 を返している。メールアドレス重複エラーが 500 として返されると、クライアントはリトライ不可の内部エラーと誤解する。

```go
// ❌ 現在のコード
res, err := c.authService.SignUp(req.Email, req.Password, req.Name)
if err != nil {
    return ctx.JSON(500, map[string]string{"error": err.Error()})
}
```

```go
// ✅ 修正案（エラー種別を判定して分岐）
res, err := c.authService.SignUp(req.Email, req.Password, req.Name)
if err != nil {
    if errors.Is(err, service.ErrEmailAlreadyExists) {
        return ctx.JSON(http.StatusConflict, map[string]string{"error": "email already registered"})
    }
    return ctx.JSON(http.StatusInternalServerError, map[string]string{"error": "internal server error"})
}
```

**理由**: Swagger に 409 を記載しているのに実装で返さないのはドキュメント詐称。クライアントが間違った再試行ロジックを組む原因となる。また `err.Error()` を直接レスポンスに含めると内部情報が漏洩するリスクもある（Supabaseのエラーメッセージが露出する場合がある）。

## 🟠 Major Issues

### [M-1] Login の成功レスポンス型が不明確

**ファイル**: `controller/auth_controller.go:54`
**問題**: `@Success 200 {object} map[string]string` となっているが、ログイン成功時の実際のレスポンス（JWTトークン等）の構造がドキュメントに反映されていない。SignUp と同様に `LoginResponse` 型を定義すべき。

```go
// ❌ 現在のgodoc
// @Success      200   {object}  map[string]string
```

```go
// ✅ 修正案
// response/auth_response.go に追加
type LoginResponse struct {
    AccessToken  string       `json:"access_token"`
    TokenType    string       `json:"token_type"`
    ExpiresIn    int          `json:"expires_in"`
    RefreshToken string       `json:"refresh_token"`
    User         UserResponse `json:"user"`
}

// controller/auth_controller.go godocを修正
// @Success      200   {object}  response.LoginResponse
```

**理由**: `map[string]string` は Swagger UI でフィールドが表示されないため、APIドキュメントとして機能しない。フロントエンド実装者が型を推測できず開発効率が落ちる。

### [M-2] `UpdateRoadmapRequest.Content` がSwaggerで型なし

**ファイル**: `requests/roadmap_request.go:10` / `docs/docs.go:1395`
**問題**: `Content interface{}` がSwagger定義で `"content": {}` (空オブジェクト＝型情報なし) になっている。APIドキュメントとして情報が欠落している。

```go
// ❌ 現在のコード
type UpdateRoadmapRequest struct {
    Title   string      `json:"title"   validate:"required" example:"ECサイト開発ロードマップ v2"`
    Content interface{} `json:"content"`
}
```

```go
// ✅ 修正案（contentの具体的な型かswaggerコメントを追加）
type UpdateRoadmapRequest struct {
    Title   string          `json:"title"   validate:"required" example:"ECサイト開発ロードマップ v2"`
    Content json.RawMessage `json:"content" swaggertype:"object"` // または具体的な構造体
}
```

**理由**: `interface{}` はGo内部では柔軟だが、Swaggerとして `{}` は意味がなく利用者に混乱を与える。ロードマップのcontentが何を意図するか明確にすべき。

## 🟡 Minor Issues

### [mn-1] BearerAuth必須エンドポイントに `@Failure 403` がない

**ファイル**: 全controller（`team_controller.go`, `roadmap_controller.go`, `requirement_controller.go`, `user_controller.go`）
**問題**: `@Security BearerAuth` が設定されているエンドポイントはすべて、権限不足時に 403 を返す可能性があるが、godocに記載がない。

```go
// ❌ 現在のgodoc（例: team_controller.go:19-31）
// @Failure      401   {object}  map[string]string
// @Security     BearerAuth
```

```go
// ✅ 修正案
// @Failure      401   {object}  map[string]string
// @Failure      403   {object}  map[string]string  "権限がない場合"
// @Security     BearerAuth
```

**理由**: 401（未認証）と403（認証済みだが権限なし）は別概念。ABACミドルウェアが実装されているため403が返される可能性がある。

### [mn-2] レスポンス定義の日時フィールドに `format: date-time` がない

**対応方法**: response構造体のフィールドにswaggoタグを追加:

```go
type TeamResponse struct {
    ID        string    `json:"id"`
    Name      string    `json:"name"`
    CreatedBy string    `json:"created_by"`
    CreatedAt time.Time `json:"created_at" example:"2024-01-01T00:00:00Z"`
}
```
