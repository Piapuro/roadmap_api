## 🔴 Critical Issues

### [C-1] `sql.ErrNoRows` を 500 として返している

**ファイル**: `controller/user_controller.go:42-45`
**問題**: `GetMySkills` でユーザーが存在しない場合（`sql.ErrNoRows`）、原因を区別せず 500 を返す。JWTは有効でもユーザーレコードが未作成のケース（Webhookの遅延など）で起こりうる。

```go
// ❌ 現在のコード
user, skills, err := c.userService.GetMySkills(ctx.Request().Context(), userID)
if err != nil {
    return ctx.JSON(http.StatusInternalServerError, map[string]string{"error": err.Error()})
}
```

```go
// ✅ 修正案
import (
    "database/sql"
    "errors"
)

user, skills, err := c.userService.GetMySkills(ctx.Request().Context(), userID)
if err != nil {
    if errors.Is(err, sql.ErrNoRows) {
        return ctx.JSON(http.StatusNotFound, map[string]string{"error": "user not found"})
    }
    return ctx.JSON(http.StatusInternalServerError, map[string]string{"error": err.Error()})
}
```

**理由**: `sql.ErrNoRows` はプログラムエラーではなくデータ不在の正常系。500を返すとクライアントが誤って「サーバー障害」と判断する。また `adapter/user_adapter.go:25` で `%w` でラップされているため `errors.Is` で到達できる。

## 🟠 Major Issues

### [M-1] `validate` タグが機能していない（バリデーション未実行）

**ファイル**: `requests/skill_request.go:4,10` / `controller/user_controller.go:82-85`
**問題**: `validate:"required,max=30"` 等のタグが定義されているが、`ctx.Bind()` はバリデーションを実行しない。Echoで実際にバリデーションするには `ctx.Validate()` の呼び出しが必要。現状では `skill_name` が空文字でも `skill_level` が不正値でも通り抜ける。

```go
// ❌ 現在のコード（Bindのみ）
var req requests.UpsertSkillsRequest
if err := ctx.Bind(&req); err != nil {
    return ctx.JSON(http.StatusBadRequest, map[string]string{"error": err.Error()})
}
```

```go
// ✅ 修正案（Bind + Validate）
var req requests.UpsertSkillsRequest
if err := ctx.Bind(&req); err != nil {
    return ctx.JSON(http.StatusBadRequest, map[string]string{"error": err.Error()})
}
if err := ctx.Validate(&req); err != nil {
    return ctx.JSON(http.StatusBadRequest, map[string]string{"error": err.Error()})
}
```

**前提**: `echo.Validator` インターフェースを実装した Validator を Echo に登録する必要がある（`go-playground/validator` が一般的）。プロジェクト全体でまだ未設定であれば、設定と合わせて対応が必要。

## 🟢 Good Points

### [G-1] トランザクションの defer Rollback パターンが正しい

**ファイル**: `adapter/user_adapter.go:41-45`
`defer tx.Rollback()` を `BeginTx` の直後に置くことで、途中でエラーリターンしてもロールバックが保証される。`Commit()` 後の `Rollback()` は no-op なので二重ロールバックの心配もない。Goのイディオムとして模範的な実装。

```go
tx, err := a.db.BeginTx(ctx, nil)
if err != nil { ... }
defer tx.Rollback()  // ← どのパスでも安全
// ... 処理 ...
return tx.Commit()
```

## 🔵 Suggestions

### [S-1] `UpsertSkills` のN回 INSERT をバルクINSERTへ

**ファイル**: `adapter/user_adapter.go:53-69`
現在はスキルを1件ずつ INSERT しているが、`unnest` を使えば1クエリにまとめられる。スキル数が多い場合のパフォーマンス改善になる。

```sql
-- sql/queries/users.sql
-- name: BulkCreateUserSkills :many
INSERT INTO user_skills (user_id, skill_name, experience_years, is_learning_goal)
SELECT $1, unnest($2::text[]), unnest($3::text[]), unnest($4::bool[])
RETURNING *;
```

現在の件数規模（~30件）では必須ではないが、将来的な選択肢として有効。

`adapter.SkillInput` と `requests.SkillInput` はフィールドが完全に同じで、service 層でのマッピングが冗長。adapter が `requests` パッケージを直接参照することには依存方向の懸念があるが、専用の `domain` / `model` パッケージを設けてそこに共通型を置く設計も検討できる。

```

```
